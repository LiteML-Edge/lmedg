# Versioning and Run Selection

This guide explains how output files are versioned and how the most recent run is selected in the preserved LiteML-Edge workflow.

## 1. Versioning convention

When a stage materializes outputs for the first time, the destination is typically created as `run/`. When the same stage is executed again, the workflow preserves earlier outputs and versions the destination as `run.v001`, `run.v002`, and so on, rather than silently overwriting prior artifacts.

## 2. Explicit evidence preserved in this package

The package includes direct training-side examples of this behavior through:

- preserved `run/` directories under `core_source_reference/trainings/`
- preserved `latest.txt` selector markers
- preserved `manifest.json` files inside saved run folders
- the shared helper implementation in `core_source_reference/utils/global_utils/versioning.py`

These materials make the versioning convention and latest-run resolution directly inspectable without requiring full workflow execution.

## 3. How metrics fits into the same logic

In the LiteML-Edge workflow, the corresponding `metrics/.../run*` directories are generated by execution of the training-side stages themselves, including the training, pruning, quantization, and downstream helper flow they drive.

Accordingly, the package does not need to duplicate a full metrics workspace to establish that the same versioning and run-selection behavior applies there. Instead, it preserves:

- the training-side versioning evidence
- the shared selector helper logic
- the pipeline descriptors that use `--selector latest` and `--runs-base`
- the downstream curated result artifacts generated from those workflows

## 4. Practical inspection path

To inspect this behavior in the package, review:

- `core_source_reference/trainings/*/*/run/`
- `core_source_reference/trainings/*/*/latest.txt`
- `core_source_reference/trainings/*/*/run/manifest.json`
- `core_source_reference/utils/global_utils/versioning.py`
- `pipelines/*.yaml`

This combination is sufficient to inspect the intended versioning convention and latest-run selector behavior without requiring the package to mirror the full generated `metrics/.../run*` tree.
